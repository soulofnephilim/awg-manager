# Changelog

## [Unreleased]

### Новые возможности

- **Соединения: редизайн страницы** — донаты по туннелям/назначению/клиентам, фильтры с чипами, группировка по клиенту/хосту, панель деталей и сброс отдельного соединения.

- **Интеграция FreeTurn** — TURN-туннель для обхода блокировок ([free-turn-proxy](https://github.com/samosvalishe/free-turn-proxy)): клиент на роутере, сервер на VPS, управление из веб-панели (новый пункт меню FreeTurn, вкладки «Клиент»/«Сервер»). Полный набор флагов клиента и сервера (провайдер VK с несколькими ссылками-пулами, streams-per-cred, ручная капча с подсказкой по SSH-пробросу, профили обфускации rtpopus/rtpopus2/rtpopus3), живые логи процессов прямо в панели, генератор и декодер share-ссылок `freeturn://` (с опциональным вложением WireGuard-конфига и автосозданием туннеля при импорте). Бинари freeturn НЕ поставляются в пакете awg-manager — панель предлагает установку в один клик: загрузка закреплённой в билде версии из релизов апстрима с проверкой SHA256 (та же модель доверия, что у managed sing-box); для архитектур без закреплённой сборки остаётся подсказка по ручной установке.

- **IPv6 endpoint для NativeWG-туннелей через awg_proxy** — сервер AmneziaWG за IPv6-адресом теперь работает в проксирующем режиме (прошивки без нативного ASC). Модуль `awg_proxy.ko` понимает бракетированную форму эндпоинта `[2001:db8::1]:51820` в `/proc/awg_proxy/{add,del,list}`, создаёт per-family UDP-сокет и отправляет трафик через `udp_tunnel6_xmit_skb` (обязательная UDP-контрольная сумма, hoplimit из маршрута; DF-бита в IPv6 нет). Локальная сторона (клиент WireGuard → `127.0.0.1:порт`) по-прежнему IPv4-loopback. Двухстековые хосты по-прежнему предпочитают IPv4 (стабильность), IPv6-only домены резолвятся в IPv6. В форме туннеля IPv6-литерал указывается в квадратных скобках (подсказка в поле Endpoint; голый `2001:db8::1:51820` отклоняется с понятной ошибкой). **Требуется пересборка `awg_proxy.ko` ≥ v1.3.0** — на старом модуле IPv6-эндпоинт отклоняется с явной ошибкой о необходимой версии, а не создаёт битый слот; при отсутствии активных слотов демон сам перегружает устаревший модуль на новый (при активных туннелях обновление откладывается до простоя). IPv6 endpoint поддержан именно для пути через awg_proxy (kmod); нативный ASC-путь NDMS (прошивки ≥ 5.01) — вне объёма: формат endpoint в RCI не менялся, поведение нативного пути не затронуто и с IPv6 не проверялось.

- **Эксперт-редактор конфигурации sing-box** — пользовательский слот `90-user.json` с черновиками, валидацией (кросс-слот проверка тегов + `sing-box check` над снапшотом «как если бы применили») и подсветкой синтаксиса; правки не перезаписываются генераторами — ни один продюсер в слот не пишет, а авточистка висячих ссылок селекторов его намеренно пропускает. В эксперт-режиме sb-router появилась кнопка «Редактор»: обзор всех слотов config.d с бейджами владельца («генерируется автоматически» / «пользовательский»), read-only просмотр системных слотов с подсветкой и копированием, для user-слота — полноценный редактор (номера строк, маркер строки с ошибкой JSON, Tab-отступы): Сохранить черновик → Проверить → Применить (debounced SIGHUP; изменение tun-inbound перезапустит движок) / Отменить черновик, плюс выключатель слота целиком без удаления содержимого. Массивы user-слота (outbounds/inbounds/правила) дополняют merged-конфиг последними; скаляры dns/route (final, strategy) отсюда не переопределяются (first-file-wins). Если применённый user-конфиг после удаления чужого outbound начинает ссылаться на несуществующий тег, reload пропускается (движок работает на старом конфиге), а причина всплывает в статусе роутера: «Пользовательский конфиг (90-user.json) ссылается на несуществующий тег … — правьте в редакторе конфигурации».

- **Импорт настроек VPN через JSON-файл mieru** ([#449](https://github.com/hoaxisr/awg-manager/issues/449)) — панели (RIXXX / Panel-Naive-Mieru) экспортируют настройки mieru каноническим JSON-конфигом клиента (формат `mieru apply config`), который раньше отклонялся всеми путями импорта. Теперь такой JSON принимается целиком — вставкой и загрузкой файла: в мастере добавления («Один сервер» и «Группа серверов» получили дроп-зону для .json/.txt/.yaml файлов) и в подписках (inline и по URL, включая превью выбора серверов). Учитывается `activeProfile`, лишние клиентские поля (rpcPort, socks5Port, loggingLevel и добавки панелей) игнорируются; TCP/UDP-привязки портов и portRange профиля дают те же outbounds, что и ссылка `mieru://`.

- **Панель Inbounds показывает все inbound'ы merged-конфига** — раньше вкладка «Inbounds» (fakeip) и панель «Inbounds» (tproxy expert) отражали только tun-in и device-proxy, хотя merged-конфиг содержит также mixed-входы подписок (`sub-*-in`), сводных групп (`agg-*-in`), туннелей и tproxy-входы QoS — в config-preview они были видны, но ни одна вкладка их не объясняла. Новый endpoint `GET /api/singbox/inbounds` читает слоты config.d по одному и атрибутирует каждый inbound источнику (подписка/группа/туннель/device-proxy/QoS/движок) с именем владельца; обе панели показывают полный сгруппированный список (управление device-proxy не изменилось), а входы, которые никто не питает (NDMS-прокси выключен или объект отключён), помечаются бейджем «резерв порта» с пояснением — порт сохраняется, чтобы при включении номера портов не менялись.

- **Regex-фильтры серверов подписки и сводные группы** ([#372](https://github.com/hoaxisr/awg-manager/issues/372)) — у подписки появились фильтры в стиле mihomo `filter:`: «включать только» и «исключать» (Go RE2, матч по имени сервера; lookahead не поддерживается — вместо `(?i)^(?!.*(RU)).*$` используется поле «исключать» с `(?i)(RU)`). Скрытые фильтром серверы не материализуются в sing-box и показаны отдельным блоком «Скрыто фильтром»; фильтры компонуются с ручным исключением, смена фильтра сразу пере-применяет подписку (для inline — повторный разбор сохранённого списка). Плюс сводные группы: один selector/urltest-туннель поверх серверов НЕСКОЛЬКИХ подписок с собственным фильтром, режимом (по умолчанию — авто-выбор быстрейшего), портом и NDMS Proxy; группы пересобираются в том же reload, что и любое изменение подписок, управление — в секции «Сводные группы» на вкладке подписок. Надёжность: стадия батч-коммита конфига сериализована между конкурентными операциями (плановый refresh больше не смешивается с CRUD подписок/групп в общем батче), тела запросов subscription-эндпоинтов ограничены 1 МБ через общий helper, сторы подписок и групп пишутся copy-on-write — при ошибке записи на диск память не расходится с файлом.

- **Настраиваемое время жизни сессии и вход по учётным данным Entware** ([#441](https://github.com/hoaxisr/awg-manager/issues/441)) — время жизни auth-сессии задаётся в настройках (`sessionTtlHours`, 1-720 часов, по умолчанию 24; скользящее окно, укорочение действует сразу на уже выданные сессии). Опциональный вход по системным учётным данным Entware (`entwareAuthEnabled`): пароль проверяется локально по `/opt/etc/shadow` (MD5-crypt/SHA-256/SHA-512-crypt, только stdlib) без обращения к NDMS `/auth` — роутер больше не шлёт уведомления о входе; при неудаче локальной проверки выполняется прежний вход через Keenetic. Защита от перебора: локальная проверка Entware добавляет поверхность офлайн-подбора, поэтому учитывается каждая неудачная попытка, в которой проверялись учётные данные (в том числе локальная Entware-проверка при недоступном роутере) — на каждую такую попытку начисляется задержка 300 мс, а после 5 с одного IP endpoint отвечает 429 в течение 30 секунд; учёт атомарный, параллельные запросы не обходят лимит. Недоступность роутера без проверки учётных данных не учитывается; ответы «пользователь не найден/заблокирован» выровнены по времени с неверным паролем (защита от перечисления пользователей).

- **QoS-классы по DSCP в sb-router** ([#371](https://github.com/hoaxisr/awg-manager/issues/371)) — трафик, помеченный DSCP-кодами (0-63, до 8 классов), направляется в отдельные выходы sing-box: классификация через iptables `-m dscp` в цепочках AWGM (per-class TPROXY/REDIRECT-порты 51281+/51301+), пары inbound'ов `tproxy-qos-N`/`redirect-qos-N` и управляемые правила маршрутизации по `inbound`. Классы редактируются в настройках sb-router (`qosClasses`); доступность xt_dscp отражается в статусе (`xtDscpAvailable`), при отсутствии модуля ядра или расширения iptables функция мягко отключается, не ломая движок.

### Улучшения

- **Рестарт и обновление awg-manager больше не перезапускают sing-box** — демон «переподключается» к уже работающему процессу: логи sing-box переезжают с пайпов на файлы в tmpfs (осиротевший процесс больше не умирает от SIGPIPE при первой записи в лог), а сверка конфигурации пропускает перезагрузку, когда применённый конфиг не изменился (раньше рестарт демона всегда дёргал sing-box, в режиме FakeIP — с полным перезапуском туннеля). Если конфиг изменился (например, после обновления) — выполняется обычная перезагрузка. VPN-соединения при рестарте/апгрейде панели больше не мигают.

- **Страницы «Туннели» и «Серверы» загружаются со скелетоном** — вместо спиннера на холодной загрузке сразу рисуется структура будущего контента (правильный режим отображения и число карточек с прошлого визита), которая заполняется данными по мере ответа роутера. Анимация уважает prefers-reduced-motion; скелет списка соединений в диагностике переведён на тот же общий стиль.

- **Страница соединений больше не грузит роутер на каждый опрос** — атрибуция правил (badge DNS-списков) читала у NDMS весь runtime-кэш object-group (сотни килобайт JSON, ~1 секунда работы ndm) на каждое обновление списка соединений: под открытой страницей это давало ~40% CPU awg-manager и постоянную фоновую нагрузку на RCI роутера. Теперь результат кэшируется на 30 секунд (данные меняются только когда DNS-правила резолвят новые IP — на свежести бейджей это не сказывается). Заодно ускорен разбор всех ответов RCI: проверка на конверт ошибки больше не парсит целиком большие успешные ответы.

- **Каталог наборов sb-router: пометки про композитные списки и наборы без правил** ([#450](https://github.com/hoaxisr/awg-manager/issues/450)) — сервисы, уже входящие в добавленный композитный список (например, Anthropic при добавленном «Все AI сервисы»), помечаются в каталоге угловым знаком и подписью «в {название}» с подсказкой «Уже входит в добавленный композитный список … Можно добавить и отдельно» — плитка при этом остаётся кликабельной (отдельный набор добавить по-прежнему можно). Задизейбленные плитки стали внятнее: «добавлено» теперь различает набор, на который ссылается хотя бы одно правило (route или DNS), и «добавлено, без правил» (добавлен как набор — не используется ни одним правилом); работает и в каталоге tproxy-эксперта, и на вкладке Rule sets режима FakeIP. В мастере правил (каталог сервисов) пресеты, чьи наборы уже есть в конфиге, получают недизейблящую пометку «набор уже есть — будет создано только правило»: мастер переиспользует существующий набор и дублей не создаёт.

### Прочее

- **sing-box** — бамп до 1.14.0-alpha.39 (совместимость с обновлённым mieru-патчем из [#459](https://github.com/hoaxisr/awg-manager/pull/459); xhttp-патч применяется без изменений). Смоук сгенерированных конфигов (tproxy и fakeip-tun, все слоты config.d) против нового бинаря: `sing-box check` и полный старт проходят чисто; наши генераторы НЕ используют конструкции, удалённые в 1.14 (legacy address filter в DNS-правилах, outbound DNS rule item, `store_rdrc`, `independent_cache` не эмитятся). Осталось два деприкейшена с удалением в 1.16 (только WARN в логе, миграция отдельным issue): `download_detour` у remote rule-set (→ `http_client`) и неявный HTTP-клиент через default outbound у remote rule-set без detour.

### Исправления

- **Карточки sing-box-подписок больше не перемешиваются** ([#525](https://github.com/hoaxisr/awg-manager/issues/525)) — список подписок отдавался в случайном порядке (итерация по map), и карточки на странице «Туннели» перепрыгивали при каждом фоновом обновлении (~раз в 30 секунд) и при перезагрузке страницы — можно было удалить не ту подписку. Теперь порядок стабильный: по названию без учёта регистра.

- **DNS-маршрутизация: включение NDMS-списка снова доезжает до роутера** ([#489](https://github.com/hoaxisr/awg-manager/issues/489)) — у списков с фолбэком «авто» выключение срабатывало, а включение оставалось только в интерфейсе AWG Manager: маршрут в прошивке продолжал висеть выключенным. Причин было две. Значение «авто» не восстанавливается из состояния роутера (NDMS не отличает его от «нет фолбэка»), из-за чего сверка считала такие маршруты вечно изменёнными и на каждом цикле пересоздавала их вместо штатного переключения флага — теперь «авто» нормализуется при сравнении, и включение/выключение идёт нативным `dns-proxy route disable` по index. Вторая: NDMS при пересоздании существующего маршрута сохраняет его флаг disable (проверено на 5.1.1) — теперь после любого пересоздания сверка перечитывает состояние и явно доводит флаг до нужного в обе стороны.

- **FakeIP: смена DNS-сервера, стека и UDP-таймаута применяется сразу** — сохранение этих полей в настройках sb-router на включённом режиме FakeIP не доезжало до движка (даже после рестарта демона) и срабатывало только при выключении/включении режима. Теперь сохранение перегенерирует fakeip-конфигурацию и перезагружает движок. Пулы адресов и MTU по-прежнему применяются при перевключении режима — они требуют пересоздания интерфейса и маршрутов.

- **Ошибки старта sing-box видны в журнале приложения** — FATAL-строки из stderr sing-box (ошибки старта, когда Clash API ещё не поднялся и штатный пересыл логов их не застаёт) теперь попадают в журнал (`/logs`, раздел sing-box), а не только в stderr самого демона.

- **Файловые логи процесса sing-box не растут бесконечно** — лог-файлы на tmpfs (`/var/run/awg-manager/sing-box.*.log`) усекаются при достижении 4 МБ; на debug-уровне при месяцах аптайма они больше не съедают tmpfs роутера.

- **FakeIP: восстановлено разрешение трафика в tun-интерфейс средствами NDMS** — при интеграции режима FakeIP из прототипа потерялась настройка firewall на OpkgTun-интерфейсе: там, где межсегментный трафик режется файрволом роутера (isolate-private и подобные политики), клиентский трафик и DNS-запросы не доходили до fakeip-интерфейса. Теперь при включении режима создаётся permit-all access-list `_WEBADMIN_OpkgTunN` с привязкой `ip access-group … in` и флагом auto-delete — ровно та же форма, которую генерирует веб-интерфейс Keenetic при разрешении доступа к интерфейсу (проверено вживую: permit → bind → auto-delete, каскадное удаление списка вместе с интерфейсом). Список снимается при выключении режима, откате неудачного включения и уборке сирот; после обновления awg-manager поверх уже включённого FakeIP список доставляется автоматически первой сверкой. ACL-команды сведены в единые NDMS-примитивы (ими же пользуется межсегментный доступ managed-серверов) — заодно ошибки этих операций, которые NDMS кладёт внутрь ответа, перестали молча проглатываться: неудачное правило доступа теперь честно всплывает, а не выглядит применённым.

- **Осиротевший интерфейс FakeIP больше не вгоняет роутер в вечный цикл перезапуска nginx** — если NDMS-интерфейс OpkgTun с настроенным `ip address` оставался после выключения FakeIP (сбой удаления, крэш посреди включения, fail-safe выключение), ndm ежесекундно пересобирал конфигурацию nginx (bind на несуществующий адрес → ошибка → регенерация → снова ошибка), постоянно съедая ~20% ядра и подвешивая ВЕСЬ RCI на секунды — веб-панель и все операции awg-manager начинали спорадически тормозить на 1–9 секунд (частая причина «awg-manager стал очень медленным» и warn'ов `perf: slow rci`). Теперь на всех путях отключения (штатное выключение, откат неудачного включения, reap) при провале удаления интерфейса его IP-адреса снимаются отдельным шагом — неудалённый интерфейс больше не оставляет опасный адрес; reap дополнен сканом по описанию (убирает осиротевшие OpkgTun'ы FakeIP, потерявшие учётную запись, вместе с их маршрутом пула) и выполняется не только на старте, но и на каждом цикле сверки — runtime-сирота лечится за полминуты без перезагрузки. Откат неудачного включения FakeIP доезжает до NDMS даже при обрыве клиента посреди операции.

- **Соединения: исправлены двойные badge DNS-списка и потеря соединений без исходящего интерфейса** — записи sing-box/tproxy (в том числе IPv6) без исходящего интерфейса в conntrack раньше терялись из списка вместо атрибуции по маршруту.

- **Rule-set'ы каталога переехали на зеркало repo.hoaxisr.ru** — списки vernette/rulesets раздавались через raw.githubusercontent.com, заблокированный у части российских провайдеров: sing-box молча не мог скачать remote-набор, и правила маршрутизации/DNS с ним не работали (частая причина «каталог не работает», в т.ч. по мотивам [#534](https://github.com/hoaxisr/awg-manager/issues/534)). Теперь 19 URL каталога указывают на зеркало `repo.hoaxisr.ru/rulesets` (источник — форк hoaxisr/rulesets с ежедневным синком upstream), существующие установки мигрируются автоматически при старте: URL в применённых sing-box-конфигах (все слоты) и DNS-подписках переписываются на зеркало без переприменения пресетов. В данные Discord добавлены подсеть `104.29.0.0/16` и голосовые порты `19200:19400` (в применённые DNS-списки Discord подсеть дописывается той же стартовой миграцией).

- **NativeWG: маска подсети из адреса интерфейса больше не заменяется на /32** ([#531](https://github.com/hoaxisr/awg-manager/issues/531)) — указанный в туннеле CIDR (например `172.25.1.2/24`) молча превращался в `/32`: при создании интерфейса маска была захардкожена, а при каждом старте синк адреса срезал суффикс до разбора маски — и перетирал даже ручную правку в веб-интерфейсе Keenetic. Теперь маска пользователя доезжает до NDMS (connected-маршрут на туннельную подсеть для LAN-to-LAN работает); адрес без маски по-прежнему получает `/32`. Помните: маска интерфейса задаёт только маршрут — что пропустит WireGuard, решает AllowedIPs пира.

- **Движок маршрутизации больше не выключается сам из-за недоступного RCI** ([#523](https://github.com/hoaxisr/awg-manager/issues/523)) — периодическая сверка трактовала ЛЮБУЮ ошибку чтения метки политики (включая транзиентную недоступность RCI при ранней загрузке или shutdown-гонке во время перезагрузки роутера) как «политика удалена» и молча делала fail-safe выключение: снос перехвата, парковка слота, `enabled=false` без записи в журнал и без авто-восстановления — после перезагрузки тумблер оказывался в «выключено». Теперь fail-safe срабатывает только когда NDMS ответил и политики действительно нет; транзиентная ошибка оставляет состояние нетронутым и ретраится следующим тиком. Причина fail-safe пишется в журнал, а каждое выключение движка (включая по кнопке) оставляет запись в журнале. Закрыт и смежный тупик: если слот конфигурации движка запаркован (rollback неудачного включения), а цепочки перехвата живы (netfilter.d-hook восстановил их из сохранённых правил), сверка считала состояние здоровым «installed» и бесконечно ждала watchdog, которому нечего чинить — теперь запаркованный слот распознаётся как дрейф и лечится полным включением (перепромоут слота + переустановка перехвата); при мёртвом процессе лечение откладывается на тик после его оживления watchdog'ом, чтобы не задерживать fail-closed блокировку. Тот же дрейф закрыт и в режиме FakeIP: запаркованный слот конфигурации возвращается в merged-конфиг и при живом движке (раньше — только при мёртвом).

- **Импорт mierus://-ссылок с дополнительными портами/диапазонами** ([#516](https://github.com/hoaxisr/awg-manager/issues/516)) — ссылка вида `...&port=2012&port=2012-2022&protocol=TCP` (один `protocol` на несколько `port`, так экспортируют панели) отклонялась с ошибкой «mismatched number of port and protocol parameters»; теперь единственный `protocol` применяется ко всем портам. Заодно исправлен собственный экспорт «Копировать ссылку»: он генерировал именно такой неканоничный формат, и ссылка не импортировалась обратно — теперь `protocol` пишется парой к каждому `port`, как в эталонном клиенте mieru.

- **NativeWG-туннели больше не делят лимит 7 ID с kernel-AWG туннелями** — раньше оба backend'а получали ID из общего пула `awg10..awg16` (7 слотов), хотя потолок 16 — прошивочное ограничение индекса OpkgTun и к NativeWG отношения не имеет (его NDMS-интерфейс — `WireguardN`, в OpkgTun не отображается). NativeWG-туннели вымещали kernel-AWG из дефицитного диапазона и сами не могли превысить 7 даже в одиночку. Теперь у NativeWG собственный неограниченный сверху диапазон ID (`awg20+`), а реальную ёмкость задают честные пределы: до 100 интерфейсов Wireguard в NDMS (индексы `Wireguard0..99`, ранее искусственно ограничивались десятью) и до 16 одновременных туннелей с обфускацией через awg_proxy (слоты kmod; предел срабатывает при СТАРТЕ туннеля на прошивках без нативного ASC и сопровождается понятным сообщением, создание интерфейсов он не ограничивает). Лимит kernel-AWG (7 туннелей, `OpkgTun10..16`) — прошивочный, без изменений. Существующие NativeWG-туннели со старыми ID `awg10..awg16` продолжают работать без миграции — kernel-аллокатор просто пропускает занятые ими номера.

- **Прокси устройств с выходом через композит роутера (vpn/vpn2) при выключенном движке маршрутизации** ([#465](https://github.com/hoaxisr/awg-manager/issues/465)) — ломались и маршрутизация, и отображение. При выключении движка слот 20-router паркуется, композиты пропадают из merged-конфига, и авточистка висячих ссылок перед каждым reload молча вырезала vpn/vpn2 из селекторов device-proxy вместе с их `default` — sing-box уводил трафик в произвольный выживший член селектора, а карточка инбаунда показывала этот runtime-тег как якобы настроенный выход (последний переключённый инстанс выглядел «правильным» лишь до следующего reload). Теперь: (1) генерация слота 30 сама не эмитит ссылки на теги выключенных слотов — при недоступном композите селектор получает его default-член (нет default'а/недоступен → первый член → direct), т.е. трафик идёт по намерению пользователя, а сохранённый выбор не трогается и восстанавливается при включении движка; (2) включение/выключение движка (и смена режима tproxy/fakeip) синхронно перегенерируют слот device-proxy ДО reload, так что авточистке нечего вырезать, а если она всё же что-то режет — пишет warn, а не молчит; (3) карточки инбаундов показывают НАСТРОЕННЫЙ выход (как «О устройстве»), живой selector.now — отдельной пометкой «сейчас: …», деградация — явной пометкой «выход недоступен — через …» (новые поля `degradedOutbound`/`fallbackTag` в runtime-ответах прокси). Попутно: стартовый reconcile device-proxy перенесён после подключения каталога выходов роутера — раньше на каждой загрузке он считал композиты удалёнными и выключал ссылающиеся на них инстансы, стирая выбранный выход.

- **Валидация «как будто применён» для выключенного целевого слота** — CheckMerged/ApplyDraft теперь всегда включают целевой слот в снапшот `sing-box check`, даже если он в данный момент выключен: черновик валидируется «как будто применён и включён», в согласии с кросс-слот проверкой. Раньше применение черновика к выключенному слоту проходило мимо `sing-box check` (затрагивало и staging роутера на выключенном слоте, и user-слот эксперт-редактора). Порядок apply у user-слота изменён на «применить, потом включить»: провал валидации оставляет припаркованный слот выключенным, а его старый конфиг — нетронутым. Попутно: конфликт тега подписки с пользовательским слотом при обновлении подписки падает сразу с честной причиной («конфликт с пользовательским слотом 90-user.json: тег … уже занят — переименуйте outbound в редакторе конфигурации») вместо непрозрачной ошибки изоляции; advisory-предупреждения валидации (например, route.final задан несколькими слотами — sing-box молча оставит первый) доезжают до ответов check/apply эксперт-редактора и показываются отдельным жёлтым блоком; включённый, но содержательно пустой 90-user.json больше не удерживает и не запускает процесс sing-box.

- **sing-box в режиме маршрутизации теперь перезапускается автоматически после падения** ([#456](https://github.com/hoaxisr/awg-manager/issues/456)) — вотчдог проверял только легаси-туннели (10-tunnels.json) и при пустом списке выходил до liveness-проверки, поэтому упавший (например, убитый OOM-killer'ом) процесс в tproxy/fakeip-режиме оставался лежать до ручного «Перезагрузить». Теперь предикат перезапуска — активная работа оркестратора (router / device-proxy / подписки / туннели), а reconcile самих режимов (tproxy и fakeip) дополнительно поднимает мёртвый процесс сам (defense in depth). Против crash-loop добавлен backoff: 3 быстрых перезапуска за 10 минут — сразу, дальше экспоненциальная пауза (1→2→4 мин, кап 15 мин); сброс после «здорового» аптайма ≥5 мин или ручного управления, ручной «Перезапустить» не подавляется. Backoff применяется и к легаси-туннелям — раньше вотчдог перезапускал упавший процесс каждые 30 секунд без ограничений. Наблюдаемость: SIGKILL без stderr классифицируется по dmesg как OOM-kill с понятной причиной («роутеру не хватило памяти…»); панель движка показывает «Падений за 10 мин», причину последнего падения и до какого времени приостановлен автоперезапуск. Попутно закрыт пробел fakeip-реконсайла: drift-heal больше не воскрешает движок, остановленный пользователем вручную.

- **Пересборка selective-ipset больше не падает по фиксированному таймауту при медленной, но идущей работе, а «живую», но затянувшуюся пересборку теперь можно отменить из UI** — раньше фоновую пересборку (API `POST /rebuild` и авто-запуск при старте/reconcile) обрывал настенный дедлайн 10 минут, хотя на медленном MIPS-роутере с большим списком доменов работа честно продолжалась. Теперь продолжительность ограничивает stall-guard по прогрессу: отмена — только если 4 минуты подряд нет ни одного сигнала прогресса (смена фазы, собранный CIDR, завершённый DNS-резолв хоста/матчера, каждая ipset-команда и restore-чанк, материализация rule-set'ов и периодический тик geosite/geoip-скана), плюс абсолютный предохранитель 2 часа от «вечно живой» пересборки. Сигналы прогресса и время простоя считаются по монотонным часам — NTP-скачок посреди boot-пересборки не даёт ни ложного «нет прогресса», ни подавленного срабатывания. Сообщение об остановке различает причины и несёт последний прогресс («нет прогресса 4 мин подряд (фаза resolving, последний матчер …)»). Отмена пересборки из UI: в модальном окне пересборки появилась кнопка «Отменить пересборку» (`POST /api/singbox/router/selective/rebuild/cancel`) — ограниченный ручной выход из медленной, но живой пересборки, которую stall-guard легально не трогает; отменённый прогон завершается событием «пересборка отменена пользователем», staging-набор не подменяет живой. Попутно: подняты тесные exec-таймауты ipset-команд (restore одного чанка 60 с → 180 с, управляющие команды create/flush/swap/destroy/list 30 с → 120 с — нагруженный роутер с maxelem=262144 в них не укладывался); decompile .srs подчинён контексту пересборки и собственному таймауту 5 минут (зависший decompile больше не переживает и stall-guard, и 2-часовой предохранитель, вечно удерживая heavyop-гейт); сбой счётчика записей после успешного swap больше не рисует ложное «ipset пустой — весь трафик в WAN» (честное «записей: н/д»); пост-обработка успешной пересборки (синхронизация маршрутов + перезапуск sing-box) ограничена 10 минутами, чтобы заклинившая другая тяжёлая операция не удерживала флаг rebuilding бесконечно.

- **Rule-set'ы из geosite.dat не срабатывали на домен верхнего уровня** ([#448](https://github.com/hoaxisr/awg-manager/issues/448)) — при создании набора правил из .dat-файла все записи geosite конвертировались в `domain_suffix` с ведущей точкой (`.chatgpt.com`), который в sing-box матчит только поддомены — сам `chatgpt.com` мимо правила уходил в `final`. Конвертация переведена на типизированное извлечение v2ray-типов: RootDomain теперь даёт `domain_suffix` + точное `domain` для апекса (как в официальных конвертерах geosite→srs), Full — точный `domain`, Plain — `domain_keyword`, Regex — `domain_regex`. Скомпилированные старым кодом .srs-кэши инвалидируются автоматически (версия формата в meta.json); просмотр geo-файлов не затронут.
- **Файлы rule-set'ов не удалялись с диска** ([#448](https://github.com/hoaxisr/awg-manager/issues/448)) — удаление или переименование набора правил оставляло его артефакты (`rule-sets/inline/*.json|.srs`, `rule-sets/dat/*.json|.srs|.meta.json`) на диске навсегда. Теперь после успешного «Применить» и один раз на старте демона выполняется сборка мусора: удаляются файлы, на которые не ссылается ни активная конфигурация, ни pending-черновик (router и fakeip-слоты); отложенное удаление в черновике по-прежнему сохраняет файлы до применения, токен dat-эндпоинта и незавершённые компиляции не трогаются.
- **Клик по названию rule-set удалял первый чип** ([#446](https://github.com/hoaxisr/awg-manager/issues/446)) — в модалке DNS-правила поле «Rule sets» было обёрнуто в `<label>`: клик по любой не-интерактивной части (названию набора) браузер форвардил в первый labelable-элемент — крестик первого чипа, «случайно» удаляя его. Обёртка заменена на `div`, а ChipMultiSelect дополнительно гасит дефолтный label-форвардинг.
- **Панель дашборда туннелей на мобильных** — переключатель «Сплошной/Разделы» рисовался с наезжающими друг на друга подписями («СплошнРазделы»): кнопки сегмент-контрола сжимались ниже ширины текста без обрезки, и текст выползал на соседнюю кнопку — теперь подписи обрезаются с многоточием. Контролы панели скомпонованы компактнее (≤4 строк): тег-фильтр и действия (Экспорт) делят одну строку 50/50 вместо двух полноширинных.

- **OOM при пересборке selective-ipset на больших списках** — импорт geosite-масштаба (сотни тысяч матчеров) убивал демон по памяти: резолвер запускал по горутине на каждый матчер, а весь список доменов и IP материализовался в RAM. Пересборка переведена на потоковый конвейер (сбор правил → ограниченная очередь → фиксированный пул DNS-воркеров) — пиковое число горутин теперь ~14 вместо сотен тысяч (бенчмарк 80k матчеров: суммарные аллокации ~44MB, 200k: ~53MB, пик горутин неизменен). Введены лимиты: 100 000 уникальных доменных матчеров на пересборку и 65 536 host-маршрутов в оверлее 19-selective-routes.json — при превышении список усекается с явной пометкой «список усечён» в итоговом сообщении и в снапшоте (`truncatedMatchers`/`truncatedRoutes`), а не валит демон. Для роутеров с 200–700MB RAM (256/512MB-модели) добавлен GC-тюнинг: мягкий `GOMEMLIMIT=96MiB` + `GOGC=50` (ранее — никакого лимита; `/system/info` теперь показывает лимит вместо «Unlimited»); явно заданные переменные окружения GOGC/GOMEMLIMIT по-прежнему имеют приоритет.

## [2.10.8] - 2026-05-19

### Исправления

#### NativeWG реализация через awg_proxy.ko — серия фиксов 

На ряде серверов NativeWG туннель через `awg_proxy.ko` либо не устанавливал handshake вовсе, либо принимал первый handshake но через ~60 секунд сервер прекращал отвечать. На том же роутере и конфигурации kernel-AWG-туннель работал стабильно. Серия инкрементальных правок (kmod версии 1.1.3 → 1.1.6) приводит к байт-идентичности с reference `amneziawg-linux-kernel-module`:

- **MAC1 пересчёт на всех 4 путях handshake** (PR [#138](https://github.com/hoaxisr/awg-manager/pull/138) автор **@AYastrebov**) — `awg_proxy.ko` ранее пересчитывал MAC1 только на 2 из 4 типов handshake-пакетов (outbound init и inbound response), пропуская outbound response и inbound init. Когда сервер инициировал rekey или клиент отвечал на init, MAC1 не совпадал и пакет дропался. Спасибо Андрею за чистый референс-кросс-чек и точечный фикс. (kmod 1.1.3), именно этот PR вернул веру в возможность починить функционал.

- **HANDSHAKE_DSCP=0x88 на init/response** — reference выставляет AF41 DSCP на handshake-пакетах. Мы отправляли с TOS=0. Промежуточные узлы по пути дропали (особенно для CPS с DNS-shape контентом). (kmod 1.1.4)

- **DF=0 (Don't Fragment не установлен)** - приведено в соответствии с reference.

- **Удалён pre-ARP 0-byte UDP "probe"** — наша собственная оптимизация для прогрева ARP оказалась первым видимым пакетом от нашего источника к серверу. (kmod 1.1.6)

- **Inter-packet jitter в handshake-цикле** — наш kthread отправлял CPS+junks+init back-to-back за <1мс. Reference (через workqueue) даёт ~7-8мс на цикл. Добавлен `usleep_range(1500,2500)` между sendmsg-ами. (kmod 1.1.6)

### Под капотом (способ реализации)

- Новый mangle-фикс `AWGM-DNS-NOPOLICY` — `internal/singbox/router/iptables.go` + `lanbridges.go`. На каждом LAN-bridge с настроенным NDMS hotspot catch-all'ом эмитим правило `-m mark --mark 0x0 -m pkttype unicast --dport 53 -j MARK --set-mark <bridge-NDMS-catch-all-mark>`. Mark передаётся per-bridge — динамически читается из `_NDM_HOTSPOT_PREROUTING_MANGL` (не хардкод), что делает фикс корректным независимо от того какая именно policy назначена default в NDMS.
- `DiscoverLANBridges(ctx)` — пересечение `(NDMS hotspot mangle catch-all)` ∩ `(real Linux bridges in /sys/class/net/*/bridge)`. Это набор bridges где (a) device МОЖЕТ оказаться с mark=0 (через MAC-override) и (b) у NDMS уже есть готовая `_NDM_HOTSPOT_DNSREDIR` инфраструктура для этого mark'a. Исключает WireGuard tunnels и не-bridge интерфейсы.
- Пустой discovery (роутер без hotspot-конфига) → skip с warn'ом, install продолжается без DNS-NOPOLICY. Регрессия невозможна.
- `Reconcile` детектит изменение LAN-bridges (added/removed bridge, mark переназначен) → re-install iptables.
- netfilter.d hook расширен comment-tag scrub'ом — после NDMS reload правила DNS-NOPOLICY восстанавливаются без duplicate-accumulation.

  - Бейдж «★» на ячейке показывает, что эта цель влияет на рестарт туннеля (совпадает с настроенным pingcheck-target).
  - Клик по ячейке открывает боковую панель с графиком latency за последний час (60 точек, обновление каждые 60 сек), статистикой (avg/min/max/loss%) и последними 10 замерами.
  - Сверху страницы — статус-полоса с 4 показателями: всего туннелей, up, failed, средняя latency.
  - Auto-refresh через SSE: матрица обновляется в реальном времени без перезагрузки страницы.
- **Раздел «Диагностика»** переработан в terminal-style ленту логов с боковой панелью «здоровья».
  - **Лента логов**: новые записи появляются сверху с blinking cursor, авто-пауза при прокрутке вниз, buffer-counter «+N новых ↑» для возврата к LIVE. Click на `[scope]` или `[LEVEL]` фильтрует по нему. Right-click открывает контекстное меню (скопировать строку, скопировать сообщение, фильтр по scope, фильтр по уровню). Ctrl+F фокусирует поиск внутри страницы.
  - **Кумулятивные фильтры по уровню**: выбор `INFO+` показывает info, warn, error. Раньше работала строгая фильтрация по одному уровню.
  - **Боковая панель «Здоровье»** содержит запуск проверок системы (с подробной настройкой режима + opt-in restart-цикла), результаты последнего запуска с раскрываемым списком тестов, и DNS-проверку (свёрнутая по умолчанию).
  - Подвкладка **«Соединения»** сохранена с косметическим обновлением.
- **Новые тесты диагностики**:
  - **Расхождение времени** — сравнивает локальное время с эталонным сервером (HTTPS Date header). Расхождение >300с = FAIL, >60с = WARN. Wireguard handshake чувствителен к skew, частая причина «туннель не поднимается».
  - **rp_filter check** — для каждого работающего туннеля читает `/proc/sys/net/ipv4/conf/<iface>/rp_filter`. Strict-mode (=1) блокирует обратный трафик через VPN — частая причина «туннель up, но интернета нет».
- **Расширение страниц** — `Мониторинг` и `Диагностика` теперь занимают всю ширину экрана, без пустых полей по бокам. Свойство `width` контейнера: `narrow` (960px) для форм, `wide` (1440px) дефолт, `full` (100%) для terminal/grid.

### Изменения работы

- **restart-цикл диагностики теперь opt-in** — раньше при выборе режима «Полная» туннели автоматически перезагружались. Теперь это отдельный чекбокс «Включая restart-цикл», по умолчанию выключен. Полезно при отладке, но не сюрприз для пользователя.
- **Журнал диагностики**: SSE-перехват пропусков. При временной потере связи с роутером (сеть переключилась, бэкенд перезапустился) недополученные записи лога подтягиваются через `since`-параметр и сливаются в feed без дубликатов.
- **Параметр уровня логирования по умолчанию**: при первом открытии Диагностики берётся настроенный в системе уровень. При смене фильтра выбор сохраняется в `localStorage` и подгружается в следующих сессиях.
- **Маршрут `/pingcheck`** теперь редиректит на `/monitoring` (старые ссылки продолжают работать). Маршрут `/connections` удалён — соединения живут как подвкладка Диагностики.

- **Графики трафика на карточках туннелей** — 1-часовое окно теперь агрегируется в 60 точек по одной за минуту. Сглаженная кривая вместо «расчёски» из 360 узлов, пики сохраняются (берётся максимум на минуту).

- **Синхронизация состояния оркестратора** после изменений настроек туннеля — устранены ложные срабатывания, приводившие к ошибкам NDMS `interface has no assigned profile` при выключении ping-check через форму редактирования.

  Если в правиле DNS-маршрутизации указано несколько туннелей (primary + backup), AWG Manager
   теперь сам переключает трафик на резервный туннель, когда primary падает (определяется
  через PingCheck). При восстановлении primary трафик автоматически возвращается. В UI
  показывается toast-уведомление о переключении.

  Это обходной путь для бага в прошивках Keenetic, где встроенный механизм fallback в
  dns-proxy route работает ненадёжно.

  Комментарии в IP-маршрутах

  В правилах IP-маршрутизации теперь можно добавлять комментарии к подсетям через синтаксис
  1.2.3.4/32 !ASTelegram. Комментарии:
  - Парсятся при импорте .bat/.txt файлов (после строки маршрута)
  - Передаются в роутер через RCI (видны в нативном WebUI Keenetic)
  - Отображаются на карточке правила в виде тегов

  Идеально для импортов с группировкой по ASN или сервисам.

  Замена опросов на Server-Sent Events (SSE)

  Полная переработка обмена данными между frontend и backend:
  - Все обновления (статус туннелей, трафик, логи, мониторинг) приходят в реальном времени
  через одно SSE соединение
  - Убрано опрашивание раз в 5–30 секунд → меньше нагрузка на роутер, мгновенные обновления в
   UI
  - Графики трафика обновляются без задержек
  - При перезапуске сервиса страница автоматически перезагружается

  Orchestrator — единый мозг управления туннелями

  Полностью переписан слой управления жизненным циклом туннелей. Теперь все события

## [2.5.0] - 2026-03-16

  AWG Manager 2.5.0

  NativeWG — нативные WireGuard-туннели через ядро Keenetic

  **Главное нововведение версии. Туннели теперь могут работать через встроенный
  WireGuard-модуль Keenetic (NativeWG) вместо userspace amneziawg-go. NDMS управляет
  интерфейсом напрямую — стабильнее, быстрее, переживает перезагрузки.**

На версии ОС Кинетик 5.1+ пока есть проблемы с NativeWG. 

  - Полный lifecycle NativeWG-туннелей: создание, старт, стоп, удаление, обновление через
  NDMS
  - Выбор backend (kernel/nativewg) при импорте конфигурации
  - Proxy-режим: awg_proxy.ko обеспечивает AWG-обфускацию поверх нативного WireGuard
  - Выбор ISP-интерфейса для NativeWG (connect via)
  - Поддержка ping-check мониторинга для NativeWG-туннелей
  - Поддержка системных туннелей в статических маршрутах и политиках

  Управляемый WireGuard-сервер

  Создание и управление собственным WireGuard VPN-сервером прямо из интерфейса AWG Manager.

  - Создание сервера с автоматическим выбором свободного интерфейса
  - NAT включён по умолчанию
  - Управление клиентами: добавление, удаление, включение/отключение
  - Индивидуальные DNS-настройки для каждого клиента
  - Генерация .conf файлов и QR-кодов для клиентов
  - Настройка ASC-параметров обфускации (junk, padding, headers, signatures)
  - Захват signature-пакетов по домену
  - Статистика пиров в реальном времени (LED, трафик, handshake)

  Страница серверов

  - Просмотр всех WireGuard-серверов (управляемых и системных) на одной странице
  - Встроенный Wireguard VPN Server отображается в read-only режиме
  - Возможность перемещать системные интерфейсы между туннелями и серверами
  - Экспорт .conf для пиров (не-встроенных серверов)

  **Удаление userspace backend

  - Amneziawg-go (userspace) полностью удалён — только kernel и nativewg
  - Уменьшен размер IPK-пакета
  - Упрощена архитектура**

  Улучшения интерфейса

  - Скрытие системных туннелей с главной страницы
  - Мониторинг (ping-check) переработан: индивидуальные настройки для каждого туннеля
  - Корректное отображение трафика клиентов (зеркальные ↓↑)

  Исправления

  - Корректная очистка при удалении программы (managed server, endpoint routes для nativewg)
  - Миграция NDMS-клиента на RCI API (меньше спама в логах роутера)
  - Поддержка KN-3611 (Hopper DSL)
